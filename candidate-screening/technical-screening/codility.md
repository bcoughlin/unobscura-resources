# Codility

**Category:** Technical Screening  
**Format:** Timed coding challenges with automated scoring  
**Assessment:** ⚠️ Significant Challenges

---

## What They Do

Codility provides timed coding challenges that evaluate candidates on algorithm and data structure problems. Often includes live coding sessions where someone observes you in real-time.

---

## Why This Is Challenging

**Strict Time Limits:**
- Usually 60-90 minutes for multiple problems
- Favors speed over thoughtfulness
- Processing speed ≠ coding ability
- No accommodation for different processing styles

**Live Observation:**
- Someone may watch you code in real-time
- Creates performance anxiety
- "Think aloud" expected
- Pressure to seem confident

**Algorithm Focus:**
- Whiteboard-style problems
- Doesn't reflect real development work
- Pattern recognition over practical skills
- Favors CS degree background

**Communication Requirements:**
- Must verbalize thinking while coding
- Bottom-up thinkers need more time to explain
- Literal communication may be misunderstood
- "Soft skills" creep into technical assessment

---

## What to Expect

**Format:**
- Online coding environment
- 2-4 algorithm problems
- 60-90 minute time limit
- May include video call with observer

**Evaluation:**
- Correctness of solution
- Code efficiency (time/space complexity)
- Code quality and readability
- How you explain your approach

**Typical Problems:**
- Array/string manipulation
- Binary search variations
- Dynamic programming
- Graph/tree traversal

---

## Specific Challenges for Neurodivergent Candidates

**For Autistic Developers:**
- Literal interpretation of ambiguous problems
- Difficulty verbalizing while coding
- Anxiety under observation
- Need for complete understanding before starting

**For ADHD Developers:**
- Time pressure increases anxiety
- Multiple problems = task switching difficulty
- Hard to focus with observer watching
- Hyperfocus on one problem at expense of others

**For All:**
- "Think aloud" is unnatural for many
- Timed format discriminates against methodical processors
- No credit for thoroughness, only speed
- Real job doesn't work this way

---

## Accommodation Strategies

**BEFORE the assessment:**

"I have [condition] and process information differently. I produce high-quality code but need accommodations:
- Untimed version or extended time (1.5-2x)
- Take-home format instead of live observation
- Written explanation option instead of verbal
- Ability to reference documentation"

**DURING the assessment:**

- Comment your code thoroughly (substitute for verbal)
- It's okay to say "I need a moment to think"
- Focus on demonstrating understanding over completion
- Ask clarifying questions liberally

**AFTER the assessment:**

Send follow-up:
"I wanted to share my complete solution with more time to think through the problem. [Attach improved version with explanation]"

---

## Alternative Approaches to Propose

"Instead of Codility, could I:
- Complete a project demonstrating real-world skills?
- Submit a code review of my GitHub work?
- Do a pair programming session on a realistic problem?
- Take home a challenge and explain my approach?"

**Frame it as:**
"I want to show my actual development capabilities, which this format doesn't capture well."

---

## Red Flags

If company:
- Refuses any accommodations
- Says "everyone takes the same test"
- Won't explain what they're measuring
- Uses Codility as only assessment

**Consider:** Is this a company that will accommodate you as an employee?

---

## Tips If You Must Take It

**Prepare:**
- Practice similar problems (LeetCode, HackerRank)
- Time yourself to build tolerance
- Practice explaining verbally (even if awkward)
- Know common patterns

**During:**
- Read all problems first, pick easiest
- Comment your approach before coding
- Partial solution > no solution
- Show problem-solving even if incomplete

**Remember:**
- This doesn't measure real development ability
- Many excellent developers fail these
- It's the tool that's broken, not you

---

## Bottom Line

Codeility systematically filters out methodical, thorough developers in favor of fast pattern-matchers. It measures algorithm memorization under pressure, not real-world coding ability.
